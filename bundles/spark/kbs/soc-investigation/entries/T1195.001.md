# Supply Chain Compromise: Software Dependencies

**What it is:** Malicious code enters the environment through a compromised or typosquatted third-party software dependency (npm, PyPI, NuGet, Go module, container base image, or a trojanized vendor update).

**How it manifests:**
- Build/CI telemetry: a `npm install`/`pip install`/`go get` pulling a new or unexpected package version; postinstall/setup scripts spawning shells or network calls during build.
- Process tree: `node`, `python`, `npm`, `pip`, or a build-agent process making unexpected outbound connections, reading env vars / `~/.npmrc` / cloud-credential files, or spawning `curl`/`powershell`.
- Network: build agents or app servers beaconing to attacker infrastructure after a dependency bump; DNS lookups to exfil domains encoding stolen tokens.
- File artifacts: modified `package-lock.json`/`requirements.txt`/`go.sum`; obfuscated code in `node_modules`.

**Investigation steps (in order):**
1. Identify the suspect package + version from build logs or SBOM; check it against advisories (typosquat, dependency-confusion, known-malicious).
2. Determine blast radius — which builds/images/hosts consumed that version (CI logs, artifact registry, SBOM/dependency graph).
3. Inspect postinstall scripts and the package source for credential theft / C2.
4. Hunt for outbound connections from build agents and app hosts to the package's exfil infra.
5. Rotate any secrets the build environment exposed (cloud keys, registry tokens).
6. Pin/remove the bad version; rebuild from a clean dependency set.

**Data sources:** CI/CD build logs, artifact-registry and package-manager logs, SBOM/dependency-graph, EDR on build agents, egress network/DNS logs, secret-access audit logs.
