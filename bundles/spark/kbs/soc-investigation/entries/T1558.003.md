# Steal or Forge Kerberos Tickets: Kerberoasting

**What it is:** Adversary requests Kerberos service tickets (TGS) for accounts with SPNs, then cracks the RC4-encrypted ticket offline to recover the service account's plaintext password.

**How it manifests:**
- Auth logs: Domain Controller 4769 (TGS request) for many distinct service SPNs from one user in a short window, especially with `Ticket Encryption Type 0x17` (RC4/legacy) requested — modern clients use AES, so an RC4 request is suspicious.
- A single workstation/user enumerating SPNs (LDAP queries for `servicePrincipalName`) then requesting a burst of TGS tickets.
- Tooling signatures: Rubeus (`kerberoast`), Impacket `GetUserSPNs.py`, PowerView `Invoke-Kerberoast`, GhostPack — visible in command lines / script-block logs.
- No failed auth needed — TGS requests succeed by design; the anomaly is the volume + RC4 + breadth of SPNs.

**Investigation steps (in order):**
1. Pull DC 4769 events; group by requesting account and count distinct service names — flag any single account requesting many SPNs, prioritizing `0x17` encryption.
2. Identify the requesting host and look for preceding LDAP SPN-enumeration.
3. Search EDR/script-block logs on that host for Rubeus/Impacket/PowerView strings.
4. Enumerate which service accounts were roasted; prioritize high-privilege/weak-password ones for forced rotation.
5. Assume the targeted service-account passwords may be cracked — rotate them and hunt for their subsequent use (T1078).
6. Check for follow-on lateral movement using the compromised service accounts.

**Data sources:** Domain Controller Security log 4769 (with encryption-type + service-name fields), 4768 (TGT), LDAP/AD query logs, EDR process + script-block logs on the requesting host.
