# Exfiltration Over C2 Channel

**What it is:** Adversary steals data by sending it out over the same channel used for command-and-control, rather than a separate exfil path.

**How it manifests:**
- Network flow: a large OUTBOUND byte volume from a host to the established C2 endpoint — asymmetric upload (bytes-out >> bytes-in) breaking the host's baseline. Often follows a collection/staging spike (large archive created locally).
- Proxy logs: sustained POST/PUT traffic, or many uploads, to the C2 domain/IP; unusual content-length; transfer continuing over hours.
- File staging artifacts: a recently created `.zip`/`.7z`/`.rar`/`.cab` in `%TEMP%`/`%PUBLIC%` (T1560) just before the egress.
- Process: the same beaconing process now moving bulk data.

**Investigation steps (in order):**
1. Quantify the egress — pull NetFlow/proxy bytes-out from the host to the C2 endpoint and compare to its baseline; identify the transfer window.
2. Identify the destination (already-known C2? newly observed?) and the process driving it via EDR.
3. Find what was staged: hunt for large recently-created archives and the files that fed them (T1560/T1074).
4. Map data scope — which file shares, DB exports, or directories the actor accessed before exfil (file-access + DB audit logs).
5. Estimate data volume/sensitivity for breach assessment; preserve logs.
6. Contain the channel (block C2, isolate host) and trace back to initial access.

**Data sources:** Firewall/NetFlow (bytes-in/out per flow), proxy/web gateway (request method + size), EDR process + network events, file-access/audit logs, DLP alerts, cloud egress logs.
