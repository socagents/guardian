# Access Token Manipulation

**What it is:** Adversary duplicates, impersonates, or steals Windows access tokens (or creates a token with specified credentials) to run code in another user's or SYSTEM's security context.

**How it manifests:**
- API/behavioral telemetry: calls to `DuplicateTokenEx`, `ImpersonateLoggedOnUser`, `CreateProcessWithTokenW`, `SeDebugPrivilege`/`SeImpersonatePrivilege` enablement; tooling like Cobalt Strike `steal_token`, `make_token`, `getsystem`, Incognito, PrintSpoofer/Potato attacks.
- Process tree: a low-privilege or service process suddenly spawns children running as `NT AUTHORITY\SYSTEM` or a different user without a corresponding interactive logon.
- Logon events: 4624 type-9 (NewCredentials, `runas /netonly`-style) or a process whose token user mismatches its parent's user.
- Service-account abuse (`SeImpersonatePrivilege`) on web/SQL servers leading to SYSTEM.

**Investigation steps (in order):**
1. Identify the process running under an unexpected user context and its parent — a child SYSTEM process from a non-SYSTEM parent is the tell.
2. Check enabled privileges (`SeDebug`/`SeImpersonate`) and any 4673/4674 privilege-use events around the time.
3. Look for known token-theft tooling and command lines in EDR (`getsystem`, `steal_token`, Potato binaries, named-pipe impersonation).
4. Correlate with logon events for the impersonated account to confirm there was no legitimate interactive session.
5. Trace what the elevated process did (credential access, persistence, lateral movement).
6. Determine the original foothold account and scope.

**Data sources:** EDR process-creation with token/user fields + API-call telemetry, Windows Security 4624 (type 9), 4672/4673/4674 (special privileges), named-pipe/Sysmon events.
