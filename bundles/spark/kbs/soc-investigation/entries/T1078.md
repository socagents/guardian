# Valid Accounts

**What it is:** Adversary uses legitimate, compromised credentials (local, domain, cloud, or default) to authenticate — blending in with normal activity and evading many detections.

**How it manifests:**
- Auth logs: successful logon (4624) from an unusual source IP/ASN/geo, at an off-hours time, or impossible-travel relative to prior sessions. Type-10 (RDP) or type-3 (network) logons from atypical hosts.
- A service/disabled/dormant account suddenly authenticating; a normal user account performing admin actions it never did before.
- Cloud: IdP sign-in from a new device/location, MFA satisfied via a registered-then-abused method, new sign-in to many resources.
- Absence of malware — the "attack" is just a login, so process telemetry may be clean; the signal is in identity logs.

**Investigation steps (in order):**
1. Establish the account's baseline — usual source IPs, hosts, hours, app set — and diff the suspicious session against it.
2. Pull all 4624/4625 (and IdP sign-in) events for the account in the window; map every host/resource it touched.
3. Determine how the credential was obtained (phishing, LSASS dump, infostealer, password spray) by pivoting upstream.
4. Check for MFA changes, new device registrations, or new app consents on the account.
5. Enumerate actions taken under the account: data access, new persistence, privilege grants, lateral logons.
6. Contain: disable/reset, revoke sessions/tokens, and scope any sibling accounts from the same source IP.

**Data sources:** Windows Security logon events (4624/4625/4768/4769), IdP/SSO sign-in + audit logs (Azure AD/Okta), VPN logs, EDR for post-auth activity, cloud audit trails (CloudTrail).
