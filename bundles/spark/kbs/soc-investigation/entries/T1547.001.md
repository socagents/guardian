# Boot or Logon Autostart Execution: Registry Run Keys / Startup Folder

**What it is:** Adversary adds a Run/RunOnce registry value or a Startup-folder shortcut so a payload executes at user logon or boot.

**How it manifests:**
- Registry keys: `HKCU\Software\Microsoft\Windows\CurrentVersion\Run` / `RunOnce`, the `HKLM` equivalents, and `...\Policies\Explorer\Run`. New value pointing at `%APPDATA%`, `%TEMP%`, `%PUBLIC%`, an encoded PowerShell one-liner, `rundll32`, `mshta`, or a `.lnk`.
- Command lines: `reg add HKCU\...\Run /v <name> /d <payload> /f`; PowerShell `Set-ItemProperty -Path HKCU:\...\Run`.
- File artifacts: dropper/shortcut in `%APPDATA%\Microsoft\Windows\Start Menu\Programs\Startup`.
- EDR registry-write events tying the modification to a suspicious parent process.

**Investigation steps (in order):**
1. Enumerate current Run/RunOnce values (autoruns-style) and flag entries in user-writable paths or with encoded/obfuscated data.
2. From EDR registry telemetry, identify which process wrote the value and when — correlate with initial access.
3. Resolve the value's target binary/script; hash and assess it (signed? known-bad?).
4. Check the Startup folder for unexpected `.lnk`/scripts.
5. Confirm execution: did the referenced payload run at the last logon/boot (process-creation events)?
6. Sweep the fleet for the same value name/payload path.

**Data sources:** EDR registry-modification events, registry autostart snapshots (Autoruns/Sysmon Event 13), process-creation (`reg.exe`/PowerShell), file-creation in the Startup folder, logon events for timing.
