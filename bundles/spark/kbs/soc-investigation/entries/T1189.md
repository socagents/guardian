# Drive-by Compromise

**What it is:** A user's browser is compromised simply by visiting a website — via an exploit kit, malicious ad (malvertising), compromised legitimate site (watering hole), or fake-update/SocGholish lure.

**How it manifests:**
- Proxy logs: navigation to a legitimate site followed by redirects to an exploit-kit gate or a `*.js` injected payload; fake browser/Flash/Teams-update domains; high entropy/randomized hostnames.
- Process tree: browser (`chrome.exe`/`msedge.exe`/`firefox.exe`) spawns `wscript.exe`/`cscript.exe` (SocGholish `.js`), `powershell.exe`, `mshta.exe`, or a dropped `.exe` from the browser cache/`Downloads`.
- File artifacts: payloads written to browser cache, `%TEMP%`, or `Downloads` with recent timestamps correlated to the browse event.
- Network: post-exploitation beacon to C2 immediately after the browse.

**Investigation steps (in order):**
1. Build the browsing timeline from proxy logs around the alert — find the entry site and the redirect/injection chain.
2. In EDR, confirm the browser spawned a script host or unsigned binary (the exploitation signal).
3. Hash and detonate any dropped file; pivot the C2 IOC across the fleet.
4. Check whether the visited legitimate site was compromised (watering hole) — query other users who hit the same site.
5. Look for follow-on persistence and lateral movement on the affected host.
6. If malvertising suspected, identify the ad network/redirector domain and block.

**Data sources:** Proxy/web gateway logs (referer + redirect chains), DNS logs, EDR process-creation + file-creation events, browser history artifacts.
