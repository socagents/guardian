# Exploit Public-Facing Application

**What it is:** Adversary exploits a vulnerability in an internet-exposed application (VPN, web app, mail server, API gateway, file-transfer appliance) to gain code execution or a foothold.

**How it manifests:**
- Web/WAF logs: anomalous requests to vulnerable endpoints — path traversal (`../`), template/expression injection, deserialization payloads, SQLi, suspicious `User-Agent`, requests to known-CVE paths; bursts of 500/302/200 around the exploit.
- Process tree: the web/app service account (e.g. `w3wp.exe`, `java`, `httpd`, `nginx`, `tomcat`) spawns `cmd.exe`/`bash`/`powershell.exe`/`whoami` — a server process should rarely spawn a shell.
- File artifacts: a web shell written into the webroot (`.aspx`, `.jsp`, `.php`) immediately after the suspicious request (see T1505.003).
- Network: outbound connection from a DMZ server that normally only receives inbound.

**Investigation steps (in order):**
1. Correlate the EDR shell-spawn event time with the web/WAF access logs to find the triggering request (source IP, URI, payload).
2. Identify the app + version and map the request to a known CVE; confirm patch status.
3. Search the webroot for files created/modified after the request — hunt web shells.
4. Review the app service account's subsequent activity (commands, new files, outbound connections, credential access).
5. Scope: query WAF/web logs for the same payload/source across all exposed apps.
6. Check for lateral movement from the compromised host inward.

**Data sources:** WAF/web access logs, EDR process + file-creation events, network flow/NetFlow at the perimeter, application/appliance logs, vulnerability-scan data.

**Pivot/related:** T1505.003 (web shell), T1059 (command execution), T1078 (valid accounts harvested), T1210 (lateral movement to internal services).
