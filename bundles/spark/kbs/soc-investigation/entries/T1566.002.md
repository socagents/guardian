# Spearphishing Link

**What it is:** Phishing email containing a link that leads to credential-harvesting, a malware download, or an OAuth consent grab — rather than an attachment.

**How it manifests:**
- Proxy/web logs: user clicks a URL to a newly-registered domain, a look-alike of your SSO/M365 login, a URL-shortener, or an attacker-controlled file host. Often follows a redirect chain or sits behind a CAPTCHA / Cloudflare gate.
- Browser process (`chrome.exe`/`msedge.exe`) spawned from `OUTLOOK.EXE`, followed by a download landing in `Downloads` then executing.
- Auth events: a successful login from an unusual ASN/geo shortly after the click — credentials phished and replayed; or an Azure AD `Add app role assignment`/illicit OAuth consent event (consent phishing).
- Mail: identical URL across many recipients; embedded link mismatched display text vs href.

**Investigation steps (in order):**
1. Pull the original email and extract the true href (expand shorteners/redirects).
2. In proxy/DNS logs, determine which users actually clicked and reached the landing page (HTTP 200 vs blocked).
3. For credential-harvest pages: check IdP sign-in logs for those users — successful auth from anomalous IP/impossible-travel = compromised account; trigger forced password reset + session revoke.
4. For OAuth consent: review enterprise app grants for suspicious new app registrations and risky scopes.
5. For malware links: trace the downloaded file's hash and execution in EDR.
6. Scope the campaign via mail logs (sender, subject, domain).

**Data sources:** Proxy/web gateway logs, DNS logs, IdP/SSO sign-in logs (Azure AD/Okta), email gateway, EDR (browser + downloads), OAuth/app-consent audit logs.
