# Command and Scripting Interpreter: PowerShell

**What it is:** Adversary uses PowerShell to execute code, download payloads, run in-memory tooling, and evade defenses — a top living-off-the-land technique.

**How it manifests:**
- Command lines: `-enc`/`-EncodedCommand` (base64), `-w hidden`/`-WindowStyle Hidden`, `-nop`/`-NoProfile`, `- exec bypass`, `IEX (New-Object Net.WebClient).DownloadString(...)`, `Invoke-WebRequest`/`iwr`, `FromBase64String`, `-c`/`-Command` with concatenated/obfuscated strings.
- Process tree: PowerShell spawned by Office, `wscript`/`mshta`, a scheduled task, `services.exe`, or a parent that has no business launching it. `powershell.exe` spawning `rundll32`/`net.exe`/`nltest` is suspicious.
- Script-block logging (Event ID 4104): decoded malicious content; Module logging 4103; `powershell.exe` with a randomized window title.
- Network: PowerShell process making outbound HTTP(S) to download or beacon.

**Investigation steps (in order):**
1. Decode any `-enc` payload (base64 → UTF-16LE) and read the actual command.
2. Pull Script-Block (4104) and Module (4103) logs for the host to reconstruct full intent — these capture deobfuscated content.
3. Identify the parent process — legitimate admin tooling vs. Office/script-host (the latter is malicious).
4. Check for downloads/C2 in proxy + DNS from the PowerShell PID.
5. Determine outcome: persistence created? Credentials touched (LSASS, Mimikatz strings)? Lateral movement (`Invoke-Command`, WinRM)?
6. Scope identical command lines across the fleet.

**Data sources:** EDR process-creation (4688), PowerShell Script-Block (4104) + Module (4103) + transcription logs, proxy/DNS logs, AMSI telemetry.
