# Go Service Conventions

## Language and Tooling

- Go 1.22+ with modules enabled.
- Lint: `golangci-lint run` (config in `.golangci.yml` if present).
- Build: `go build ./...`
- Test: `go test ./... -race`

## Code Style

- Exported names use PascalCase. Internal names use camelCase.
- Always handle errors explicitly. Wrap with context: `fmt.Errorf("operation: %w", err)`.
- No blank error returns (`_ = something()`). Handle or propagate.
- Use `context.Context` as the first parameter for functions that do I/O or may block.

## Testing

- Table-driven tests for functions with multiple input/output cases.
- Test files are `*_test.go` colocated with source.
- Use `testify` for assertions if already in the project, otherwise standard `testing`.
- Target 80%+ coverage on new code.

## Project Structure

- Each service has its own `go.mod` under `services/<name>/`.
- Internal packages go in `internal/`.
- Proto-generated code goes in `gen/` or a dedicated package — do not edit generated files.

## gRPC

- All inter-service APIs use gRPC. Proto definitions live in `contracts/`.
- Implement service interfaces generated by `buf generate`.
- Use interceptors for logging, auth, and error mapping.
